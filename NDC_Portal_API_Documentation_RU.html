I. Введение

1.1 Обзор

NDC Portal APIs — Wonder Travel (версия 1.0) разработан для обеспечения бесперебойного поиска рейсов, подтверждения тарифов, бронирования и дополнительных услуг. Он позволяет туристическим агентствам и партнёрам напрямую подключаться к авиакомпаниям, обеспечивая доступ к актуальным ценам, наличию мест и расширенному контенту. NDC Portal APIs предназначены преимущественно для использования системами.


Перечень URL-шаблонов ресурсов:

  • /api/Offers/GetAvailableOffers
  • /api/Offers/FareConfirmOffer
  • /api/Offers/GetOfferAvailableBundles
  • /api/Offers/AddPassengersDetails
  • /api/Order/BookAndPay
  • /api/Order/Hold
  • /api/Order/BookAfterHold
  • /api/Order/FareConfirmAfterHold
  • /api/Order/Retrieve
  • /api/Wallet/GetAvailableBalance

Данная документация содержит исчерпывающее руководство по интеграции с API, охватывающее аутентификацию, структуры запросов/ответов, обработку ошибок, ограничения частоты запросов и сценарии использования.

1.2 Версия API

  • Текущая версия: 1.3
  • Дата выпуска: 05.08.2025
  • Статус: Стабильная

В будущих версиях будут внедрены улучшения на основе требований авиакомпаний и агентств. Убедитесь в совместимости с версией 1.1 при интеграции.

1.3 Ключевые возможности

API предоставляет широкий спектр возможностей, включая:

  • Поиск рейсов — получение актуальных предложений рейсов по заданным критериям.
  • Подтверждение тарифа — проверка выбранного тарифа перед оформлением бронирования.
  • Бронирование и выписка билетов — оформление резервирования рейса и выпуск билетов.
  • Удержание бронирования (оплата позже) — резервирование рейса без немедленной оплаты.
  • Пакеты услуг (Bundles) — дополнительные услуги: дополнительный багаж, выбор места, приоритетная посадка.
  • Управление PNR — получение деталей бронирования.
  • FareConfirmAfterHold — переоценка предложения после его удержания.
  • Баланс кошелька — проверка доступного баланса агентства перед бронированием.

Примечание: Отмена бронирования — в настоящее время отмена через API не поддерживается. Она должна обрабатываться через официальные каналы связи.

1.4 Целевая аудитория

Данная документация предназначена для:

  • Туристических агентств и онлайн-агентств (OTA), интегрирующих услуги бронирования рейсов.
  • Разработчиков и инженеров, занимающихся реализацией API.

1.5 Как использовать документацию

Для эффективной интеграции с NDC Portal APIs следуйте следующим шагам:

  • Изучите сценарии использования — начните с понимания реальных сценариев бронирования, чтобы получить чёткое представление о различных рабочих процессах.
  • Изучите аутентификацию — узнайте, как аутентифицировать запросы к API с помощью ключей API и белых списков IP.
  • Изучите цикл бронирования рейса — следуйте процессу от поиска до бронирования, включая подтверждение тарифа и выписку билетов.
  • Проверьте обработку ошибок — ознакомьтесь с кодами ошибок и их значением для эффективной обработки ответов API.
  • Изучите ограничения запросов — понимайте лимиты запросов и последствия их превышения.
  • Обратитесь к дополнительным ресурсам — найдите ссылки на коллекцию Postman и официальные каналы поддержки.

II. Аутентификация

NDC Portal APIs требует аутентификации для обеспечения безопасного доступа. Аутентификация осуществляется с помощью ключа API. Кроме того, все запросы должны исходить с IP-адресов из белого списка. Любой запрос с неодобренного IP будет отклонён.


Пример попытки несанкционированного доступа:

{ "statusCode": 401, "message": "Доступ запрещён из-за недействительного ключа подписки. Убедитесь, что предоставлен действительный ключ активной подписки." }

2.1 Метод аутентификации

Клиенты должны включать ключ API в заголовки запроса. Имя заголовка: Ocp-Apim-Subscription-Key. Пример:

curl --location 'BaseURL/Endpoint' --header 'Ocp-Apim-Subscription-Key: ВАШ_КЛЮЧ'

2.2 Требование белого списка IP-адресов

Перед получением доступа к API необходимо зарегистрировать ваши IP-адреса по соображениям безопасности. Запросы с незарегистрированных IP-адресов будут заблокированы. Запросы на добавление в белый список IP должны быть направлены по электронной почте на официальный адрес связи.

2.3 Получение учётных данных

Учётные данные для песочницы будут выданы по электронной почте на официальный адрес связи агентства после подписания формального соглашения с Wonder Travel. Учётные данные для продуктивной среды будут предоставлены по электронной почте после успешного прохождения процесса сертификации с Wonder Travel.

2.4 Пример запроса аутентификации

curl --location 'BaseURL/Api/Offers/GetAvailableOffers' --header 'Ocp-Apim-Subscription-Key: ВАШ_API_КЛЮЧ'

III. Эндпоинты

NDC Portal APIs предоставляет набор RESTful-эндпоинтов, обеспечивающих цикл бронирования рейса — от поиска доступных предложений до бронирования и управления резервированиями.

Все запросы к API должны выполняться по протоколу HTTPS и требуют аутентификации с помощью ключа API.

2.1 Общий формат

Каждый эндпоинт API имеет следующий общий формат:

  • HTTP метод: POST
  • Тело запроса: JSON-структура
  • Тело ответа: JSON-структура
  • Аутентификация: Ключ API

2.2 /api/Offers/GetAvailableOffers — Получение доступных предложений

Эндпоинт «Получение доступных предложений» позволяет агентствам получать доступные предложения на основе заданных параметров поиска.

Пример запроса

{ "searchCriteria": [ { "origin": "CAI", "destination": "RUH", "date": "2025-09-02T12:42:03.914Z" } ], "passengers": [ { "passengerTypeCode": "ADT", "count": 1 } ] }

Описание атрибутов запроса

Тело запроса:

  • searchCriteria (Критерии поиска) — данные о поездке, необходимые для поиска доступных рейсов:
    • origin — код аэропорта отправления (формат IATA, например «CAI» для Каира).
    • destination — код аэропорта назначения (формат IATA, например «JED» для Джедды).
    • date — дата поездки.
  • passengers (Пассажиры) — определяет количество и типы пассажиров:
    • passengerTypeCode — тип пассажира: ADT (взрослый), CHD (ребёнок), INF (младенец).
    • count — количество пассажиров данной категории.

Тело ответа:

  • segments (Сегменты) — определяют участки (остановки) маршрута. Каждый ключ — идентификатор сегмента. Включает: код аэропортов отправления/назначения, время вылета/прилёта, продолжительность полёта в минутах, код воздушного судна, коды оперирующего и маркетингового перевозчика.
  • journeys (Маршруты) — словарь, где каждый ключ представляет уникальный идентификатор маршрута. Содержит коды аэропортов, количество остановок, список ключей сегментов.
  • priceClasses (Классы тарифов) — словарь, сопоставляющий идентификаторы классов тарифов с деталями: название класса, тип тарифа (публичный/приватный), код тарифной группы, правила и штрафы.
  • baggageDetails (Детали багажа) — данные о багаже для маршрутов. Содержит: ручная кладь (carryOnBaggage) и зарегистрированный багаж (checkInBaggage).
  • offers (Предложения) — список объектов, определяющих доступные рейсы с ценами, сегментами, багажом, налогами, классом обслуживания и информацией о скидках. Каждое предложение уникально идентифицируется offerId:
    • passengerFareBreakdown — детализация тарифа по типу пассажира: общая сумма, налоги, базовая сумма, скидка, детали сегмента.
    • priceDetails — итоговые детали цены для всех пассажиров: общая сумма, налоги, базовая сумма, скидка, Refundability (возвратность).
    • haveBundles — флаг наличия пакетов услуг. Если true — необходим дополнительный запрос для получения пакетов.
    • canBeHeld — флаг возможности удержания бронирования. Если true — рейс можно временно зарезервировать.

Фильтры поиска API

Фильтры поиска API — Новая функция

Функция фильтров поиска позволяет клиентам уточнять и сортировать результаты поиска рейсов по нескольким критериям. Доступны следующие фильтры:

1. Фильтр класса обслуживания

"cabinClassFilters": ["ECONOMY"] // Тип: массив строк

Фильтрует результаты по классу обслуживания (например, Economy или Business). Значения должны быть допустимыми. Дублирование не допускается.

2. Фильтр RBD (Код бронирования)

"rbdFilters": ["T"] // Тип: массив строк

Фильтрует предложения по коду бронирования (RBD). Каждое значение должно быть ровно одним алфавитным символом (без дублей). Недопустимые примеры: "TT", "1", "@".

3. Фильтр предпочтительных авиакомпаний

"preferredAirlines": ["MS", "SV"]

Включает в результаты только предложения указанных авиакомпаний. Код должен состоять ровно из 2 символов. Нельзя использовать вместе с restrictedAirlines.

4. Фильтр запрещённых авиакомпаний

"restrictedAirlines": ["XX", "YY"]

Исключает предложения указанных авиакомпаний из результатов. Нельзя использовать вместе с preferredAirlines.

5. Только прямые рейсы

"directFlightsOnly": false // true = только прямые рейсы

6. Включить ближайшие аэропорты

"includeNearbyAirports": false // true = включить ближайшие аэропорты

7. Фильтр диапазона цен

"priceRange": { "minPrice": 100, // minPrice >= 0 "maxPrice": 500 // maxPrice >= minPrice }

8. Критерии сортировки

"sortingCriteria": ["FastestFirst"]
ЗначениеОписание
FastestFirstСначала самые быстрые (по общему времени в пути)
SlowestFirstСначала самые долгие
CheapestFirstСначала самые дешёвые
MostExpensiveFirstСначала самые дорогие
EarliestDepartureFirstСначала ближайший вылет
LatestDepartureFirstСначала поздний вылет

Пример запроса с фильтрами

{ "searchCriteria": [{ "origin": "DXB", "destination": "CAI", "date": "2026-01-12T08:50:37.506Z" }], "passengers": [{ "passengerTypeCode": "ADT", "count": 1 }], "searchModifiers": { "filters": { "directFlightsOnly": true, "includeNearbyAirports": true, "rbdFilters": ["T"], "priceRange": { "minPrice": 100, "maxPrice": 10000 }, "cabinClassFilters": ["ECONOMY"], "preferredAirlines": ["SM"] }, "sortingCriteria": ["FastestFirst"] } }

2.3 /api/Offers/FareConfirmOffer — Подтверждение тарифа

Этот эндпоинт позволяет агентству выбрать идентификатор предложения для получения обновлённой цены и деталей предложения.

Пример запроса

{ "offerId": "R2FsaWxlbyNFR1kjQTFDMUkx..." }

Описание атрибутов

Тело запроса:

  • offerId — ключ предложения из ответа GetAvailableOffers.

Тело ответа аналогично GetAvailableOffers, но вместо offers используется поле selectedOfferOptions.

Ключевые отличия для функции Upselling (Апсейл):

  • priceClasses — возвращает дополнительные тарифные бренды для выбранного маршрута.
  • baggageDetails — предоставляет дополнительные варианты багажа для покупки.
  • selectedOfferOptions — возвращает несколько вариантов апгрейда с соответствующими тарифными брендами и багажными услугами.

Upselling — это коммерческая стратегия авиаотрасли, направленная на улучшение впечатлений пассажира путём предложения связанных тарифных брендов и дополнительных вариантов багажа для одного и того же маршрута.

2.4 /api/Offers/GetOfferAvailableBundles — Доступные пакеты услуг

Получает описания «услуг» с ценами, варьирующимися для каждого пассажира, и идентификаторы доступных пакетов, связанных с предложением. Используется, если флаг предложения HaveBundles = true.

Примечание: В настоящее время некоторые авиакомпании требуют запрос Bundles как обязательный шаг в процессе резервирования рейса. Другие считают этот запрос необязательным. Структура ответа чётко указывает, необходим ли этот шаг для конкретного предложения.

Пример запроса

{ "OfferId": "Rmx5TmFzI0VHWSNBMUMxSTEj..." }

Пример ответа

{ "offerJourneys": { "XY~ 264~ ~~CAI~01/10/2025 10:30~RUH~01/10/2025 14:05~~": { "origin": "CAI", "destination": "RUH", "bundleReferences": ["0"] } }, "bundleChoices": { "0": { "bundleCode": "LCAI", "bundleName": "Light", "bundlePrices": [{ "passengerTypeCode": "ADT", "totalAmount": { "amount": 0.000, "currencyCode": "EGP" } }], "includedServices": [ "7 кг ручная кладь", "2 x 20 кг зарегистрированный багаж", "Предоставляется закуска", "Бесплатное изменение бронирования", "Отмена бронирования за плату" ] } } }

Описание атрибутов

Тело запроса: offerId — ключ предложения из ответа FareConfirm.

Тело ответа:

  • offerJourneys — маршруты предложения с ссылками на пакеты.
  • bundleChoices — доступные пакеты услуг:
    • bundleCode — используется для покупки пакета при бронировании.
    • bundleName — название пакета.
    • bundlePrices — цены по типам пассажиров (без младенцев): общая сумма, налоги, сборы.
    • includedServices — список услуг, включённых в пакет.

2.5 /api/Offers/AddPassengersDetails — Добавление данных пассажиров

Позволяет агентству передавать данные пассажиров после согласия с тарифом из ответа FareConfirm.

Пример запроса

{ "OfferId": "R2FsaWxlbyNFR1kjQTFDMUkx...", "Passengers": { "Pax1": { "PassengerTypeCode": "ADT", "Title": "MR", "Gender": "male", "Name": { "First": "Иван", "Middle": "", "Last": "Иванов" }, "BirthDate": "1990-06-13", "NationalityCountryCode": "RU", "ResidenceCountryCode": "RU", "TravelDocument": { "DocumentNumber": "AB123456", "DocumentType": "Passport", "ExpirationDate": "01/01/2030", "IssuanceDate": "01/01/2020", "NationalityCountryCode": "RU" }, "Contact": { "Email": "ivan@example.com", "Phone": { "CountryDialingCode": "+7", "PhoneNumber": "9001234567", "Type": "Mobile" } } } } }

Пример ответа

{ "offerId": "R2FsaWxlbyNFR1kjQTFD...ОБНОВЛЁННЫЙ_ID..." }

Описание атрибутов

Тело запроса:

  • offerId — ключ предложения из ответа FareConfirm.
  • Passengers — данные всех пассажиров:
    • title — обращение (MR, MRS и т.д.).
    • name — имя, отчество, фамилия.
    • NationalityCountryCode — гражданство пассажира.
    • ResidenceCountryCode — страна проживания.
    • birthDate — дата рождения.
    • ParentPassengerReferenceId — для INF (младенца) — ссылка на взрослого пассажира (ADT).
    • gender — пол пассажира.
    • passengerTypeCode — тип: ADT (взрослый), CHD (ребёнок), INF (младенец).
    • travelDocument — данные документа: тип, номер, дата выдачи, срок действия.
    • Contact — контактные данные: email, телефон, адрес.

Тело ответа: offerId — для следующего шага (Hold или BookAndPay).

2.6 /api/Order/BookAndPay — Бронирование и оплата

Запрос на бронирование используется для окончательного оформления покупки выбранного предложения после подтверждения тарифа (FareConfirm).

Пример запроса

{ "offerId": "Rmx5TmFzI0VHWSNBMUMxSTEj...", "selectedBundles": [ { "journeyKey": "XY~ 263~ ~~RUH~01/20/2025 07:20~CAI~01/20/2025 09:40~~", "selectedBundleCode": "LCAI" } ] }

Описание атрибутов

Тело запроса:

  • offerId — ключ из ответа AddPassengersDetails.
  • selectedBundles — детали пакета (если есть):
    • journeyKey — ключ маршрута.
    • selectedBundleCode — код выбранного пакета.

Тело ответа:

  • ndcBookingReference — NDC-ссылка на бронирование (используется для Retrieve).
  • airlinePnr — PNR авиакомпании.
  • gdsPnr — PNR системы GDS.
  • journeys — маршруты с bundleReferenceIds.
  • segments — детали сегментов рейса.
  • baggageDetails — информация о багаже.
  • includedBundles — словарь с деталями приобретённых пакетов.
  • passengers — данные пассажиров с номером билета (ticket).
  • order — детализация заказа: маршруты, разбивка тарифа, детали цены (Refundability).

*Примечание: Только NDCBookingReference используется для получения бронирования (Retrieve).

2.7 /api/Order/Hold — Удержание бронирования

Функция удержания позволяет агентству временно зарезервировать бронирование с возможностью оплаты позже. Удержание обычно истекает через один день. Функция доступна, когда флаг предложения canBeHeld = true.

Пример запроса

{ "OfferId": "A1C0I0#eca6119c-ee88-44b4-827c-...", "selectedBundles": [] }
Примечание: selectedBundles не может быть пустым, если флаг HaveBundles предложения равен true.

Описание атрибутов ответа

  • ndcBookingReference — ссылка для запроса BookAfterHold.
  • airlinePnr — PNR авиакомпании.
  • gdsPnr — PNR системы GDS.
  • HoldExpirationDate — крайний срок оплаты бронирования.
  • journeys, segments, baggageDetails, passengers, order — аналогично BookAndPay.
  • includedBundles — пустой объект при удержании без пакетов.

*Примечание: В настоящее время удержание предложений с пакетами не поддерживается.

2.8 /api/Order/BookAfterHold — Бронирование после удержания

Операция «бронирование после удержания» завершает оплату резервирования, помещённого на удержание (обычно на срок до одного дня).

Пример запроса

{ "airlinePnr": "QGSF5Y", "gdsPnr": "BXT4DM", "ndcBookingReference": "NDCEG-BR-F250613zRXWqQ", // обязательно "BookingToken": "askl5p8wf-ka25d" }

Тело запроса:

  • ndcBookingReference ОБЯЗАТЕЛЬНО — NDC-ссылка из ответа Hold.
  • airlinePnr — PNR авиакомпании.
  • gdsPnr — PNR системы GDS.
  • BookingToken — токен из ответа Hold.

Структура ответа аналогична BookAndPay, включая данные пассажиров с номером билета.

2.9 /api/Order/Retrieve — Получение данных бронирования

Получает детали существующего бронирования по PNR.

Пример запроса

{ "ndcBookingReference": "строка", "airlinePnr": "строка", "gdsPnr": "строка", "bookingToken": "строка", "leadPassengerLastName": "строка" }

Тело запроса: NDC-ссылка на бронирование из ответов Hold, BookAndPay или BookAfterHold.

Ответ содержит полные данные бронирования: маршруты, сегменты, пассажиров, пакеты, детали заказа, включая HoldExpirationDate (если применимо).

*Примечание: Только NDCBookingReference используется для получения бронирования.

2.10 /api/Order/FareConfirmAfterHold — Переоценка после удержания

Повторно проверяет цену предложения перед окончательным оформлением удержанного бронирования.

Пример запроса

{ "ndcBookingReference": "NDCEG-BR-F250528g" }

Тело запроса: NDC-ссылка на бронирование из ответа Hold.

Ответ возвращает актуальные данные: маршруты, сегменты, классы тарифов, детали багажа и обновлённый заказ с текущей ценой.

2.11 /api/Agency/AgencyBalance — Баланс агентства

Проверяет доступный баланс агентства перед инициированием бронирований или дополнительных транзакций.

Пример ответа

{ "TopupBalance": { "amount": 7858.95, "currency": "EGP" }, "WalletAmount": { "amount": 3012.10, "currency": "EGP" }, "CreditLimitBalance": { "amount": 0.0, "currency": "EGP" }, "OdeysysOs": { "amount": 10871.05, "currency": "EGP" }, "IsBlocked": false }

IV. Обработка ошибок

API следует единообразному подходу к обработке ошибок. Ошибки возвращаются в стандартизированном формате JSON, включающем: Status, CorrelationId, ErrorMessage, TimeStamp.

CorrelationId необходим при обращении в службу поддержки.

4.1 Общие коды ошибок

HTTP кодКод ошибкиОписание
200okЗапрос успешно обработан, ответ возвращён.
201createdЗапрос успешно обработан, создан новый ресурс.
204no_contentЗапрос успешно обработан, содержимое ответа отсутствует.
400bad_requestЗапрос недействителен: неверный синтаксис, отсутствующие параметры или некорректные данные.
401unauthorizedТребуется аутентификация или она не прошла из-за недействительных/отсутствующих учётных данных.
404not_foundЗапрашиваемый ресурс не найден на сервере.
409supplier_issueКонфликт из-за проблемы с системой авиакомпании или поставщика.
429Too Many RequestsПревышен лимит частоты запросов.
500internal_errorНепредвиденная ошибка на стороне сервера при обработке запроса.

Ошибка 401: Не авторизован

{ "statusCode": 401, "message": "Доступ запрещён из-за недействительного ключа подписки. Убедитесь в предоставлении действительного ключа активной подписки." }

Ошибка 400: Неверный запрос

Ошибка 409: Проблема с поставщиком

{ /* Возвращается при отсутствии доступных предложений */ "correlationId": "7cf0c315-3746-4e55-b68c-db0a2a8d2434", "statusCode": 409, "message": "Нет доступных предложений. Сохраните CorrelationId для дальнейшего расследования.", "timestamp": "2025-03-13T06:43:44.897491Z" }

4.3 Лучшие практики обработки ошибок

  • Проверяйте коды и сообщения об ошибках перед повторной попыткой.
  • Убедитесь, что все обязательные параметры включены в запросы.
  • При сохранении проблем обращайтесь в службу поддержки, предоставляя полный ответ с ошибкой для диагностики.

V. Ограничения частоты запросов

Для обеспечения справедливого использования и поддержания производительности API NDC Portal применяет ограничения частоты запросов. Эти ограничения управляются API и различаются для сред Sandbox и Production.

СредаЛимиты запросовПримечания
Sandbox (Тестирование)Стандартные лимитыСтандартное ограничение, применяемое Azure API Manager.
Production (Продуктив)ДоговорныеЛимиты определяются в процессе заключения договора с агентством.

При превышении лимита запросов API применяет следующие механизмы ограничения:

  • Ответ 429 Too Many Requests — при превышении лимита API возвращает ошибку 429.
  • Заголовок Retry-After — ответ может содержать заголовок Retry-After, указывающий время ожидания перед следующим запросом.
  • Временная блокировка — при сохранении чрезмерной нагрузки к ключу API или IP-адресу может быть применена временная блокировка.

VII. Поддержка и контакты

Для обеспечения бесперебойной интеграции и решения проблем NDC Portal APIs предоставляет несколько каналов поддержки на различных этапах разработки и эксплуатации.

7.1 Как получить поддержку

В период разработки:

  • Основной контакт: все запросы в службу поддержки направляются на официальный адрес электронной почты.
  • Еженедельные звонки (по желанию): агентства могут запросить плановые еженедельные звонки для технических обсуждений и устранения неполадок.

После запуска в эксплуатацию:

  • Основной контакт: поддержка доступна исключительно через официальный адрес электронной почты.
  • Время ответа: запросы в службу поддержки будут обработаны в рамках согласованного SLA.

7.2 Дополнительные ресурсы

  • Коллекция Postman — готовая к использованию коллекция Postman для тестирования API. Доступ предоставляется при подключении к системе.

По любым дополнительным вопросам обращайтесь по официальному адресу электронной почты поддержки.

VIII. Приложение

Данное приложение содержит дополнительные сведения и пояснения для лучшего понимания ключевых концепций NDC Portal APIs.

8.1 Общая терминология

1. NDC (New Distribution Capability — Новые возможности дистрибуции)

NDC — стандарт, разработанный IATA для модернизации авиадистрибуции. Он обеспечивает прямые связи между авиакомпаниями и туристическими агентствами, повышая прозрачность тарифов, персонализацию предложений и актуальность доступности.

2. Предложение и подтверждение тарифа

  • Предложение (Offer) — набор вариантов рейсов, возвращаемых в ответ на запрос поиска, содержащих детали тарифа, доступность и включённые услуги.
  • Подтверждение тарифа (Fare Confirmation) — процесс проверки тарифа и доступности предложения перед оформлением бронирования. Обеспечивает неизменность цены до завершения транзакции.

3. Удержание бронирования (Оплата позже)

Удержание позволяет агентству зарезервировать рейс без немедленной оплаты. Это временно фиксирует тариф и доступность мест на определённый период, после которого бронирование должно быть либо подтверждено (выписан билет), либо отменено.

4. Бронирование (Выписка билетов)

Операция бронирования — завершение процесса резервирования рейса путём выпуска билетов. Это фиксирует данные о поездке пассажира и требует подтверждения оплаты.

5. PNR (Passenger Name Record — Запись о пассажире)

PNR — уникальный референс бронирования, хранящий данные пассажира, маршрут и статус бронирования в системе авиакомпании. Генерируется при успешном создании бронирования.

  • PNR авиакомпании — референс, сгенерированный непосредственно в системе авиакомпании.
  • Системный PNR (GDS PNR) — внутренний референс туристического агентства или агрегатора рейсов.

6. Пакеты услуг (Bundles)

Пакет — набор дополнительных услуг, предлагаемых вместе с авиабилетом: багаж, выбор места, питание на борту и другие вспомогательные услуги. Для некоторых предложений выбор пакета обязателен перед завершением бронирования.

7. Дополнительные услуги (Ancillaries)

Дополнительные услуги — это услуги, приобретаемые отдельно от базового тарифа:

  • Дополнительный багаж
  • Выбор места
  • Приоритетная посадка
  • Доступ в VIP-зал аэропорта

8. Схема мест (Seat Mapping)

Схема мест обеспечивает визуальное отображение занятых и свободных мест на борту воздушного судна, позволяя пассажирам выбирать место по своим предпочтениям.

9. SSR (Special Service Request — Запрос на специальное обслуживание)

SSR — запрос на специальную помощь или дополнительные услуги, например:

  • Помощь маломобильным пассажирам
  • Специальное питание
  • Перевозка домашних животных
  • Сопровождение несовершеннолетних без сопровождающих

8.2 Рейсы на основе маршрутов и рейсы на основе предложений

NDC Portal APIs в настоящее время поддерживает модель на основе предложений, обеспечивая фиксацию цен и доступности на момент получения данных.


Что такое рейсы на основе предложений?

  • Каждый поиск рейсов возвращает список предложений, включающих детали рейса и гарантированный тариф.
  • Тариф остаётся действительным в течение ограниченного времени, снижая риск изменения цены между поиском и бронированием.
  • Используется на этапах FareConfirm и Booking для обеспечения точности цен.
  • Идеально подходит для прозрачного отображения цен на рейсы.

Рейсы на основе маршрутов (не поддерживаются)

В настоящее время рейсы на основе маршрутов, предоставляющие варианты рейсов без фиксации тарифов, не поддерживаются. Все результаты поиска возвращают финализированные предложения с подтверждёнными ценами.
